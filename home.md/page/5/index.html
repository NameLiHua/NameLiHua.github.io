相关概念

缓存击穿问题也叫热点Key问题，就是一个被高并发访问并且缓存重建业务较复杂的key突然失效了，无数的请求访问会在瞬间给数据库带来巨大的冲击。

![[Pasted image 20230705103905.png]]

解决方案

方案一：

使用锁，当一个线程发现需要重新访问SQL的时候，将获取锁，并执行更新操作，下一个线程就不能获取到这个锁，可以执行休眠操作。
![[Pasted image 20230705104233.png]]

但是这种方案对用户并不友好，因为可能不断的重试。

方案二：

使用逻辑过期处理。
其实可以将热点key设置为永不过期，但是这样的话就会一直占用内存。
使用逻辑过期的方法就可以解决key一直占用内存的问题。

**具体的解决办法:
并不真正在redis中设置过期，而是在val中设置过期时间，后续通过逻辑进行处理。

自平衡的二叉搜索树

关于红与黑-GPT的回答

在红黑树中，每个节点都有一个颜色，可以是红色或黑色。以下是红黑树的性质：

  1. 每个节点要么是红色，要么是黑色。
  2. 根节点是黑色的。
  3. 每个叶子节点（NIL节点，即空节点）是黑色的。
  4. 如果一个节点是红色的，则它的两个子节点都是黑色的。
  5. 对于每个节点，从该节点到其所有后代叶子节点的简单路径上，均包含相同数目的黑色节点（即黑色节点的数量相同）。

通过这些性质，红黑树可以保持平衡，从而保证了树的高度在一个相对较小的范围内，并且在插入和删除节点时可以保持较好的性能。

需要注意的是，红黑树的”红”和”黑”只是用来标记节点的颜色，并没有其他特殊含义。这些颜色标记是为了满足红黑树的性质而引入的。

散列表中使用到了红黑树和链表

相当于数组的变种，相当于映射。

hash函数

key存入之前需要使用hash函数进行转换，实际上存的key并不是key本来的样子。

![[Pasted image 20230717162305.png]]

hash冲突

因为当前再好的hash函数都不能保证：key1 != key2 && hash(key1) != hash(key2)
当：**key1 != key2 && hash(key1) == hash(key2) *时，hash冲突就产生了。

解决办法：拉链法

拉链法就是：当产生hash冲突的时候，公用同一个key，同时将val成链表存储。
![[Pasted image 20230717163002.png]]
这个方法可能会导致散列表退化成链表。

所以引入红黑树至散列表中，将拉链法中的链表替换为红黑树，提高查找效率
